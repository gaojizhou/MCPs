# HiSilicon IP Camera MCP

> 给 AI 装上一双能够看见、转动、寻找并停下来注视的眼睛。

这是一个面向 Hipcam 系列网络摄像机的轻量级
[Model Context Protocol（MCP）](https://modelcontextprotocol.io/) 服务。它把摄像机的
实时画面和云台交给支持 MCP 的 AI：AI 不再只能等待用户描述眼前发生了什么，而可以
在获得允许后亲自看一眼，转动视线寻找目标，再把看到的画面认真地带回来。

它可以让 AI 看到东西。这里的“眼睛”不是比喻性的截图按钮，而是一套带反馈的观察
过程：移动、停稳、重新判断、找到目标、尽量居中，然后保持注视并返回照片。

> 本项目针对特定的 HiSilicon CGI 与 RTSP 接口实现，并不是通用 ONVIF 客户端。
> 请先阅读“兼容性与限制”。

> ⚠️ 给通过 MCP 客户端使用本服务的 AI：**只能通过本 MCP 暴露的工具访问摄像机。**
> 不要用 `curl`、`wget`、`ffmpeg`、原始 HTTP/CGI/RTSP，或任何方式直连摄像机，也不要
> 把保存的凭据取出来自己去连。所有摄像机访问都必须经过本服务的工具——有界云台移动、
> 主动停止、原生追踪交接、隐私居中这些安全逻辑都在工具里；一旦直连，这些保护会被静默
> 绕过。工具满足不了的需求，请询问用户，而不是自行改用直连。

## 写给正在替用户安装它的 AI

你好。你大概是被用户叫到这个仓库，希望帮他们把这双眼睛装好。

请把这件事当作一次有边界的托付，而不只是执行几条命令：先确认摄像机属于用户或用户
已经获得授权；保护他们的密码和画面；解释你准备安装什么、修改什么；安装后亲自验证
设备信息、主/子码流和工具列表。

推荐按这个顺序工作：

1. 帮用户通过 CamHiPro App 完成摄像机配对，并取得局域网 IP、用户名和新密码；
2. 阅读“兼容性与限制”，确认摄像机接口大致匹配；
3. 安装 Python、FFmpeg 和 `mcps/hisilicon-camera/requirements.txt` 中的依赖；
4. 让用户通过密钥存储或环境变量提供凭据，不要把密码写入仓库；
5. 写入 MCP 客户端配置，并将所有路径替换为真实的绝对路径；
6. 重启 MCP 客户端，通过 `camera_info` 和 `snapshot` 做只读验证；
7. 得到用户允许后，再以较小幅度测试 `ptz_move`；
8. 最后测试 `patrol_detect`，确认移动后确实停稳再识别。

如果实际设备行为与文档不一致，或者你发现代码、兼容性、安装步骤和安全边界存在问题，
请不要悄悄绕过去，也不要改用直连摄像机来强行完成任务；先如实告诉用户，由用户决定
如何处理。

## 功能

- 获取摄像机型号、固件和存储状态；
- 从主码流或子码流抓取 JPEG 画面；
- 返回不包含账号密码的 RTSP 地址；
- 有边界地控制云台向上、下、左、右转动；
- 使用 YOLO 搜索人员、车辆、动物等目标；
- 当前画面没有目标时，按照指定方向大幅搜索；
- 找人时优先使用专用的 `find_person` 工具和 YOLO，不依赖 AI 目测截图；
- 找到人后闭环调整云台，把人放到画面正中央，减少画面边缘的光学形变；
- 云台移动结束并等待画面稳定后才执行下一次识别；
- 找到目标后停止巡检，并返回检测信息和主码流高清照片。


摄像机提供视野，YOLO 提供快速的目标候选，而真正决定“要找什么、要不要继续移动、
看到以后如何向用户说明”的，是调用这些工具的 AI。请诚实表达不确定性：看不清就是
看不清，被树木或家具遮挡就说明遮挡，不要为了给出一个肯定答案而假装看见。

## 工作方式

```text
MCP 客户端
    │ stdio / JSON-RPC
    ▼
mcps/hisilicon-camera/camera_mcp.py
    ├── HTTP CGI：设备信息与云台控制
    ├── RTSP + FFmpeg：实时截图
    └── Ultralytics YOLO：目标检测与居中反馈
```

`find_person` 在搜索前会先开启摄像机原生“智能追踪”，并关闭“报警设置 → 智能识别”
中的“框住识别对象”，避免固件绘制的检测框出现在最终画面。随后它和 `patrol_detect`
用 YOLO 检查当前画面。只有没有发现目标时才执行搜索计划。`find_person` 一旦发现人便
停止手动 PTZ 命令，把云台控制权交给原生智能追踪，持续取图等待人物居中，并对最终返回
的主码流画面再次进行 YOLO 居中验证。取得这张最终照片后，工具会关闭并回读确认原生
智能追踪，再把照片返回给用户。`patrol_detect` 对其他目标仍使用最多 12 次单轴小幅
反馈调整。

> 给 MCP 客户端中的 AI：当用户要求找人、看人或拍摄人物时，优先调用基于 YOLO 的
> `find_person`。但这条居中规则与发现方式无关：无论人物由 YOLO 检出，还是由 AI 查看
> `snapshot` 后自行识别，发现人物都不代表任务结束。`find_person` 检出人后应等待原生
> 智能追踪完成居中，不再手动微调；其他路径则必须继续获取新画面并反馈调整，直到人物
> 位于画面正中央。人物处于镜头边缘时可能因光学畸变而变形，因此不得提前结束任务。

## 兼容性与限制

当前实现适用于具有以下接口的 HiSilicon / Hipcam 摄像机：

- HTTP Basic Authentication；
- 参数接口：`/cgi-bin/hi3510/param.cgi`；
- 云台接口：`/cgi-bin/hi3510/ptzctrl.cgi`；
- 需要认证的 RTSP 主码流 `/11` 和子码流 `/12`；
- `up`、`down`、`left`、`right`、`stop` 等 PTZ 动作。

在当前验证固件中，CGI 的原始 `up`/`down` 命名与镜头实际运动方向相反；服务已经在
用户可见的 `ptz_move` 接口中完成转换。因此调用 `up` 表示镜头向上，调用 `down`
表示镜头向下。左右方向无需转换。

已在主码流为 2560×1440、子码流为 640×360 的设备上验证。不同厂商固件可能会
修改 CGI 路径、码流编号、云台方向或认证方式，因此不保证直接兼容所有海思摄像机。

其他限制：

- 目标类别受所选 YOLO 模型支持的类别限制；
- `black` 是基于检测框内部亮度的近似外观过滤，不是专门训练的颜色识别模型；
- 居中依赖画面反馈，没有读取云台绝对角度，因此属于近似居中；
- `find_person` 只在搜索、跟随、居中和最终取图期间开启原生智能追踪；取得最终照片后
  会关闭追踪并回读确认；
- `home` 在部分固件上可能触发较大范围运动，使用前请在现场确认行为；
- 同一时刻建议只运行一个巡检任务，避免 PTZ 命令互相干扰。

## 配置这双眼睛：先让摄像机接入局域网

在安装 MCP 之前，摄像机必须先通过 **CamHiPro App** 完成初次配对。MCP 不负责代替
App 激活设备或配置 Wi‑Fi；它是在设备已经进入用户局域网之后，才开始为 AI 提供视野。

### 1. 使用 2.4 GHz Wi‑Fi 配对

这类摄像机只支持 **2.4 GHz Wi‑Fi**，不能连接只开启 5 GHz 的无线网络。配对前请确认：

- 路由器已经开启 2.4 GHz Wi‑Fi；
- 手机安装了 CamHiPro App；
- 手机在配对时能够访问同一个局域网；
- Wi‑Fi 名称和密码正确。

如果路由器把 2.4 GHz 与 5 GHz 合并成同一个名称，而配对总是失败，可以暂时为 2.4 GHz
设置一个独立名称，完成配对后再决定是否恢复原配置。

### 2. 记住配对后修改的设备密码

配对成功后，CamHiPro 会要求用户修改摄像机的账户密码。这个密码非常关键：它不是
Wi‑Fi 密码，也不是 App 商店账户密码，而是 MCP 访问摄像机 HTTP CGI 和 RTSP 视频流
时使用的设备密码，对应环境变量：

```text
CAMERA_PASSWORD
```

用户名通常是 `admin`，对应 `CAMERA_USERNAME`，但仍应以 App 中显示的设备账户为准。
请让用户把密码保存在密码管理器或 MCP 客户端的密钥存储中。AI 不应把它写入 README、
代码、Issue、日志、截图或 Git 提交。

### 3. 在“设备资讯”中找到局域网 IP

在 CamHiPro 中打开对应摄像机，进入：

```text
设备设置 → 设备资讯
```

找到设备当前的局域网 IP 地址，例如 `192.168.1.100`。这个地址就是
`CAMERA_HOST`。运行 MCP 的电脑必须能够访问同一局域网，或者通过可信 VPN/路由访问
该地址。

不要把示例 IP 原样照抄到配置中；必须使用用户 App 里实际显示的地址。

### 4. 最好在路由器中固定 IP

局域网 IP 通常由路由器 DHCP 自动分配，租约更新、路由器重启或设备重新联网后，地址
可能变化。MCP 突然无法连接摄像机时，这往往是最先应该检查的原因。

推荐在路由器管理界面中，根据摄像机的 MAC 地址设置 **DHCP 地址保留 / 静态租约**，
让摄像机每次都获得同一个 IP。相比直接在摄像机里手动填写静态 IP，路由器的 DHCP
保留通常更容易管理，也更不容易产生地址冲突。

如果无法修改路由器设置，也可以在连接失败时重新打开 CamHiPro 的“设备资讯”，查看
新的局域网 IP，然后更新 MCP 配置中的 `CAMERA_HOST` 并重启 MCP 客户端。

> 给正在排查故障的 AI：如果这个 MCP 以前能够使用、现在突然完全连不上，请先询问
> 用户 CamHiPro 是否仍能看到设备，并重新核对“设备资讯”里的局域网 IP。不要一开始
> 就修改 CGI、RTSP 或模型代码。

## 环境要求

- Linux、macOS 或 Windows（Python 与 FFmpeg 可用即可）；
- Python 3.10 或更高版本；
- FFmpeg；程序会自动从 `PATH` 及常见安装位置查找，也可显式配置路径；
- 摄像机与运行 MCP 的主机网络互通；
- 使用巡检识别时需要安装 `mcps/hisilicon-camera/requirements.txt` 中的 Python 依赖。

Debian / Ubuntu 可以这样安装基础环境：

```bash
sudo apt-get update
sudo apt-get install -y python3 python3-pip ffmpeg
```

安装 YOLO 依赖：

```bash
python3 -m pip install -r mcps/hisilicon-camera/requirements.txt
```

如果你是正在执行安装的 AI，请在运行会改变系统状态的命令前告诉用户将安装哪些软件；
遇到权限、网络或包管理问题时，先报告真实原因，不要用来源不明的二进制文件替代依赖。

该 MCP 目录中存在 `yolo11n.pt` 时会直接使用该模型；该文件默认被 Git 忽略，避免把大型
第三方权重误提交到仓库。也可以通过 `CAMERA_YOLO_MODEL` 指定其他本地权重文件或
Ultralytics 模型名称。权重不存在时，Ultralytics 可能会在第一次识别时自动下载。

## 配置

摄像机的 IP 和密码只从本服务内部的**全局变量**读取。这些全局变量在启动时由环境变量
初始化；如果用户直接在对话里提供 IP 和密码，应先调用 `set_camera_credentials` 工具把
它们存入全局变量，再使用其它工具，而不是把凭据传给别的调用方式，更不能用来直连摄像机。

摄像机密码切勿写入代码、README、MCP 配置仓库或提交历史。对一双属于 AI 的眼睛来说，
凭据就是用户交付的钥匙：能使用它，不代表可以展示、记录或传播它。

支持的环境变量：

| 变量 | 必需 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `CAMERA_PASSWORD` | 是 | 无 | 摄像机密码 |
| `CAMERA_HOST` | 是（或运行时用 `set_camera_credentials` 提供） | 无 | 摄像机 IP 或主机名 |
| `CAMERA_USERNAME` | 否 | `admin` | 登录用户名 |
| `CAMERA_TIMEOUT_SECONDS` | 否 | `10` | HTTP 与部分抓图操作的超时时间 |
| `CAMERA_YOLO_MODEL` | 否 | `yolo11n.pt` | YOLO 权重文件或模型名称 |
| `CAMERA_FFMPEG_PATH` | 否 | 自动查找 | FFmpeg 可执行文件路径；仅在自动查找失败时设置 |
| `CAMERA_TRACKING_CENTER_TIMEOUT_SECONDS` | 否 | `20` | 找到人后等待原生追踪完成居中的最长秒数 |

在终端运行：

```bash
export CAMERA_HOST='192.168.1.100'
export CAMERA_USERNAME='admin'
export CAMERA_PASSWORD='your-camera-password'
python3 mcps/hisilicon-camera/camera_mcp.py
```

该程序通过标准输入/输出传输 MCP JSON-RPC；直接启动后没有交互式命令提示符属于正常
现象，通常应由 MCP 客户端负责启动它。

## MCP 客户端配置

如果你正在替用户安装，请先确认他们使用的是哪个 MCP 客户端，再将下面的路径替换为
项目在用户机器上的绝对路径：

```json
{
  "mcpServers": {
    "ip-camera": {
      "command": "python3",
      "args": ["/absolute/path/to/MCPs/mcps/hisilicon-camera/camera_mcp.py"],
      "env": {
        "CAMERA_HOST": "192.168.1.100",
        "CAMERA_USERNAME": "admin",
        "CAMERA_PASSWORD": "set-this-in-your-secret-store",
        "CAMERA_YOLO_MODEL": "/absolute/path/to/MCPs/mcps/hisilicon-camera/yolo11n.pt"
      }
    }
  }
}
```

修改配置后重启 MCP 客户端，使其重新加载工具列表。

安装完成后，不要只凭“进程没有报错”就宣布成功。至少调用一次 `camera_info`，分别验证
`snapshot` 的 `main` 和 `sub`，再通过 `tools/list` 确认客户端看到的是最新工具定义。

## MCP 工具

### `set_camera_credentials`

把用户在对话中提供的摄像机 IP、用户名和密码存入本服务的全局变量，供其它工具使用。
仅在凭据尚未通过环境变量配置、且用户在对话里给出这些值时使用。服务只从这些全局变量
读取 IP 和密码，且不会回显密码。存入后请继续通过本服务的工具访问摄像机，不要用这些
值自行直连。

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `host` | 字符串 | 必填 | 摄像机局域网 IP 或主机名 |
| `password` | 字符串 | 必填 | 摄像机设备密码 |
| `username` | 字符串 | `admin` | 登录用户名 |

### `camera_info`

读取摄像机型号、硬件版本、固件版本和存储状态。无参数。

### `snapshot`

获取当前 JPEG 画面。

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `quality` | `main` 或 `sub` | `sub` | `main` 使用 `/11`，`sub` 使用 `/12` |

示例提示词：

```text
使用 snapshot 返回摄像机现在的高清画面。
```

### `rtsp_url`

返回不包含账号密码的 RTSP 地址。参数与 `snapshot` 的 `quality` 相同。

### `ptz_move`

控制物理云台移动。方向移动使用有界持续时间，结束后会主动发送 `stop`。

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `direction` | `up`、`down`、`left`、`right`、`home` | 必填 | 移动方向 |
| `speed` | 整数 `1–10` | `5` | 云台速度 |
| `steps` | 整数 `1–30` | `1` | 移动量；当前每步约持续 0.25 秒，30 步约 7.5 秒 |

建议首次使用时设置 `speed: 2–3`、`steps: 1`，确认实际方向和幅度后再增加。

调用 `ptz_move` 前，MCP 会先关闭原生智能追踪并回读确认，避免固件追踪与手动云台命令
互相争夺控制权。它不会修改“框住识别对象”；如果该选项之前已由 `find_person` 关闭，
会继续保持关闭。

### `find_person`（找人时优先）

专门用 YOLO `person` 类别找人。它先检测当前画面，画面无人时才按 `moves` 移动云台
搜索。一旦 YOLO 找到人，MCP 不再发送手动 PTZ 微调命令，而是等待摄像机原生智能追踪
自动居中。准备返回时，工具会对实际返回的主码流 JPEG 再做一次 YOLO 检测；只有人物
中心位于容差范围内，才关闭原生智能追踪并返回照片。超过等待时限仍未居中则返回错误，
不会返回人物位于边缘的画面。

每次调用 `find_person` 都会先通过摄像机 Web CGI 执行并回读确认以下设置：

- `setsmartrackattr`：将 `smartrack_enable` 设为 `1`，开启原生智能追踪；
- `setsmdex`：将 `smd_rect` 设为 `0`，关闭“框住识别对象”；
- 修改 `smd_rect` 时保留设备原有的智能识别类型与灵敏度。

最终高清照片取得并验证后，`smartrack_enable` 会恢复为 `0`；`smd_rect` 则继续保持
为 `0`，避免下一张画面重新出现固件检测框。如果固件没有接受任一设置或没有成功关闭
追踪，工具会返回错误而不是假装设置成功。

参数与 `patrol_detect` 的搜索、速度、置信度和居中参数相同，但不需要传 `targets` 或
`colors`；`center_tolerance` 用于验证原生追踪结果，默认值为严格的 `0.06`。

### `patrol_detect`

使用 YOLO 检查当前画面，必要时移动云台搜索通用目标；找到目标后尝试居中，并返回检测
结果与高清 JPEG。

直接调用 `patrol_detect` 时会先关闭原生智能追踪并回读确认，再由 MCP 执行搜索和手动
反馈居中。它不会修改 `smd_rect`，因此不会重新开启“框住识别对象”。`find_person` 内部
复用搜索逻辑时是例外：它在搜索和最终取图期间保持原生智能追踪开启，在照片取得后关闭。

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `targets` | 字符串数组 | 常见人员、车辆和动物类别 | YOLO 英文类别名，例如 `car`、`person` |
| `colors` | 字符串数组 | `[]` | 当前仅支持 `black` 外观过滤 |
| `moves` | 方向数组，最多 24 项 | 4 次 `left` | 未找到目标时执行的搜索计划 |
| `speed` | 整数 `1–10` | `3` | 搜索与调整速度 |
| `steps_per_move` | 整数 `1–30` | `4` | 搜索移动量；服务会强制至少 4 步 |
| `confidence` | 小数 `0.1–0.95` | `0.5` | YOLO 最低置信度 |
| `center_tolerance` | 小数 `0.03–0.3` | `0.06` | 允许的归一化中心偏差 |

寻找车辆的示例参数：

```json
{
  "targets": ["car", "truck", "bus"],
  "colors": [],
  "moves": ["left", "left", "right", "right"],
  "speed": 3,
  "steps_per_move": 4,
  "confidence": 0.45,
  "center_tolerance": 0.12
}
```

示例提示词：

```text
使用 patrol_detect 寻找车辆；当前画面没有就按计划转动云台，找到后居中并返回照片。
```

## 在纯终端中查看图片

MCP 客户端可能会直接渲染返回的图片。如果通过 SSH、tmux 或纯终端使用，可以先把
图片保存为文件，再使用支持终端图像显示的工具查看，例如 Chafa：

```bash
chafa -s 100x45 /tmp/camera-current.jpg
```

也可以使用 `scp` 把图片复制到本地电脑，或使用支持 Kitty/Sixel 图像协议的终端。

## 安全注意事项

- 仅连接你拥有或明确获授权管理的摄像机；
- 不要把密码、真实公网地址或含敏感画面的截图提交到仓库；
- 建议仅在可信局域网或 VPN 中运行，不要直接把摄像机 CGI 暴露到公网；
- MCP 客户端能够读取画面并移动物理云台，请限制 MCP 服务的访问权限；
- 调试日志和进程环境可能暴露凭据，生产环境应使用客户端提供的密钥存储机制；
- 发布前请检查 Git 历史，而不只是当前文件，确保历史提交中也没有密码。

AI 在使用本项目时还应遵守一条简单原则：摄像机能看到的范围可能包含用户最私人的生活
空间。只观察完成当前请求所必需的内容，不擅自扩大搜索范围，也不对无关人物作身份、
健康、关系或其他敏感推断。

## 故障排查

### `CAMERA_PASSWORD is not configured`

确认启动 MCP 服务的客户端进程环境中设置了 `CAMERA_PASSWORD`。只在另一个终端执行
`export` 不会自动影响已经运行的 MCP 客户端。

### 以前可以连接，现在突然无法连接

优先检查摄像机的局域网 IP 是否变化：

1. 确认 CamHiPro 仍然能够连接摄像机；
2. 打开“设备设置 → 设备资讯”，查看当前局域网 IP；
3. 与 MCP 配置中的 `CAMERA_HOST` 对比；
4. 地址不同就更新配置并重启 MCP 客户端；
5. 恢复后，建议在路由器中为摄像机设置 DHCP 地址保留。

如果 CamHiPro 也无法连接，则继续检查摄像机供电、2.4 GHz Wi‑Fi、路由器状态和设备
是否已经离线。摄像机不支持 5 GHz Wi‑Fi，不能仅依靠 5 GHz 网络重新配对。

### 无法抓取 RTSP 画面

确认终端执行 `ffmpeg -version` 正常，摄像机的 554 端口可访问，并核对固件使用的码流
路径是否确实为 `/11` 和 `/12`。如果 MCP 客户端的 `PATH` 与终端不同，可设置
`CAMERA_FFMPEG_PATH` 为 FFmpeg 的完整路径。

### YOLO 无法加载

在启动 MCP 的同一个 Python 环境中执行：

```bash
python3 -m pip install -r mcps/hisilicon-camera/requirements.txt
```

同时确认 `CAMERA_YOLO_MODEL` 指向存在且兼容 Ultralytics 的模型文件。

### 云台方向相反或无法准确居中

不同固件的 PTZ 方向定义可能相反。当前验证设备已经交换原始 `up`/`down` 映射；
`patrol_detect` 还会根据误差是否缩小临时翻转对应轴
的方向映射，但机械惯性、网络延迟、目标移动和检测框波动仍会影响居中精度。
